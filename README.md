# zoter-Google-doc-toolkit

Make Zotero citations in Google Docs faster, smarter, and a bit more fun.

A Google Apps Script add-on that lets you write inline placeholders like
`[@zotero:ABCD1234]` or `[@zotero:find]` while drafting, then rewrite
them into Zotero's own [transfer-document format][internals] in one
click. The Zotero Connector then ingests them on its next **Refresh**,
exactly as if you'd added them through the normal Zotero UI.

## Two tag families

### Resolve known citations

When you already know what you want to cite:

```
[@zotero:ABCD1234]              Zotero item key (8 chars)           ✓ works
[@zotero:10.1038/nature12373]   DOI (must be in your library)       ✓ works
[@zotero:pmid:29028643]         PubMed ID                           ✓ works
[@zotero:?free text query]      Search your Zotero library by title ✓ works
[@zotero:240618027Automated…]   Better BibTeX citekey (Extra field) ⚠ see below
[@zotero:ABCD1234|p. 42]        Add a page locator                  ✓ works
```

**Known issue — Better BibTeX citekeys.** Unique citekeys generated by
the [Better BibTeX][bbt] plugin (e.g. `smithCrisprAnalysis2024` or
`240618027AutomatedClinical`) are **not yet reliable**. BBT stores the
citekey in each item's Extra field, which the Zotero Web API's `q`
parameter does *not* search — we work around this by iterating the
library and regex-matching the Extra field locally, but the Extra field
format varies across BBT versions (`Citation Key:`, `tex.citationkey:`,
`Citekey:`, …) and some items don't write it at all. If a citekey tag
fails, fall back to the DOI or the 8-char Zotero item key (both always
work).

### Find new citations

When you have a claim and want evidence AND a counterpoint:

```
[@zotero:find]
```

Each `[@zotero:find]` tag is replaced **inline** with three
peer-reviewed references, each hyperlinked to its DOI or URL:

```
(Smith 2022 [orig]; Brown 2021 [rev]; Jones 2023 [contra])
```

- **orig** — original/primary research paper that **supports** the claim
- **rev** — review or meta-analysis that **supports** the claim
- **contra** — paper that **questions, contradicts, or warns against**
  the claim — forced into every result so you get a balanced counterpoint

The sidebar shows a card per tag with the three refs, their titles,
justifications, and the exact Claude API cost for that call. A running
session total is displayed next to the "Find" heading.

These are **research suggestions** the user should review — not
finalized Zotero citations. If you decide to keep one, click its
hyperlink to open the paper, save it to Zotero via the Connector, then
rewrite as a `[@zotero:DOI]` or `[@zotero:KEY]` tag and run Resolve.

**Cost reality check.** `[@zotero:find]` runs Claude Sonnet 4.6 with
Anthropic's server-side `web_search_20260209` tool and dynamic filtering.
A thorough Find run over a dozen or so tags in a paper-length draft
**realistically costs around $1 in API credits**. Don't be stingy —
cutting the token budget degrades the quality of the counterpoint
search, which is the whole point. The sidebar shows the per-call cost
and session total so you can track spend as you go; if you want to
reduce cost, do fewer Find calls, not cheaper ones.

## Install

Open [`INSTALL.md`](INSTALL.md). The fast path is **paste-based** — no
command line, no Node, no `clasp`. About 10 minutes. End-user paste
blocks live in [`dist/INSTALL_PASTE.md`](dist/INSTALL_PASTE.md).

## Safety

- Uses the `documents.currentonly` OAuth scope. The script can only see
  the Doc that is currently open — never any other doc on your Drive.
- API keys live in `PropertiesService.getUserProperties()`, scoped per
  user per script.
- One install covers every Google Doc on your account, present and
  future. No per-doc reinstall, no per-doc credentials.

### Roadmap

- **Auto-fix corrupted Zotero Google Docs** *(planned — high priority)*.
  The Zotero connector occasionally leaves docs in an inconsistent state
  (orphaned NamedRanges, broken `?broken=<key>` hyperlinks, partially
  applied Refresh mutations). Today the recovery requires manual
  triage. A planned sidebar tool will detect and auto-repair the common
  breakage patterns using the same NamedRange / hyperlink rules
  documented in `docs/ZOTERO_INTEGRATION_INTERNALS.md`. This is the
  single highest-impact feature we haven't shipped yet — corrupted docs
  have ruined real days of writing.
- **Reliable Better BibTeX citekey support** — needs either a stable
  Extra-field format convention across BBT versions, or a local
  `localhost:23119/better-bibtex/json-rpc` bridge (not reachable from
  Apps Script alone).
- **Reliable auto-open sidebar on doc load**. The current `onOpen`
  hook calls `showSidebar()` but in practice it doesn't fire under
  Test Deployment + "Installed for current user" — a known Apps
  Script quirk where the sidebar show is silently suppressed until
  a user-initiated menu click. Open the sidebar once from
  **Extensions → My Happy Zotero Toolkit → Open sidebar**; once open
  it stays open across edits. A real fix probably needs Marketplace
  publication or a different trigger mechanism.
- **Footnote-mode citations** (`noteType: 1` in doc prefs).
- **Optional auto-save** of discovered papers from `[@zotero:find]`
  into the user's Zotero library (requires write-access API key).
